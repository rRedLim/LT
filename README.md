# Lenta Tech Life Hack — «Полка под контролем»

Решение для автоматического распознавания ценников с видеопотока робота,
движущегося вдоль стеллажей. На вход — видео торгового зала, на выход —
структурированный **CSV в формате организаторов (29 колонок)**, где каждая
строка соответствует одному уникальному обнаруженному ценнику.

```
видео.mp4  ──►  [ веб-интерфейс ]  ──►  ML-пайплайн  ──►  result.csv (скачать)
```

## Архитектура

Два слоя:

1. **`mlcore/`** — боевой ML-пайплайн (обучение моделей + inference).
   Точка входа inference: `mlcore/src/inference/pipeline.py::run()`,
   `video.mp4 → CSV`. Подробности по обучению/воспроизведению моделей —
   в `mlcore/README.md`.
2. **`webapp/`** — front + back на **Gradio** (один процесс): загрузка
   видео, запуск пайплайна, отдача CSV + превью + сводка + лог. Тяжёлые
   импорты (torch/transformers/ultralytics) — ленивые, UI поднимается сразу.

```
┌─────────────────────────── Docker (CUDA) ───────────────────────────┐
│                                                                      │
│   Браузер ──HTTP:7860──►  Gradio (webapp/app.py)                      │
│                               │                                      │
│                               ▼                                      │
│                  mlcore/src/inference/pipeline.run()                 │
│         YOLO11s+ByteTrack → Qwen2.5-VL-7B+LoRA → barcode → CSV        │
│                               │                                      │
│   volumes:  ./models:/models:ro   hf-cache   ./outputs               │
└──────────────────────────────────────────────────────────────────────┘
```

Решение работает **в локальном контуре** — без обращения к облачным API.
Базовая VLM скачивается с HuggingFace один раз в кэш-volume.

## Логика пайплайна

| Этап | Что делает | Технология |
|------|------------|------------|
| 1. Детекция + трекинг | Находит ценники на каждом кадре, склеивает в треки (1 трек = 1 физический ценник) | YOLO11s + кастомный ByteTrack (длинный `track_buffer` против motion blur) |
| 2. Выбор кадра | Из трека берёт самый резкий, не обрезанный рамкой кадр | Laplacian sharpness (`pick_A`) |
| 3. Чтение полей | Извлекает 23 поля ценника в JSON | Qwen2.5-VL-7B + LoRA-адаптер, dynamic resolution + gallery-match названий |
| 4. Штрихкод | Декод по всем кадрам трека (28 препроцессингов × 4 поворота), при пустом — VLM-fallback | zxing-cpp |
| 5. Маршрутизация | Расставляет `нет` / `""` по архетипу ценника | template-router (`archetype_schema.yaml`) |
| 6. Запись | CSV строго в формате организаторов (29 колонок, локаль-разделители) | `csv_writer` |

«Быстрый (smoke)» режим в UI выполняет только этапы 1–2 + структуру CSV —
полезен для проверки I/O и демонстрации без нагрузки на VLM.

## Быстрый старт (Docker, GPU)

**Требования на сервере:** Docker + Docker Compose, NVIDIA-драйвер,
[NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html),
GPU с ~20+ ГБ VRAM (для Qwen2.5-VL-7B в bf16).

```bash
# 1. Конфигурация (опционально — есть рабочие дефолты)
cp .env.example .env

# 2. Положить веса (см. models/README.md):
#    models/yolo/best.pt          — детектор
#    models/lora/final/           — LoRA-адаптер (папка)

# 3. Сборка и запуск
docker compose up -d --build

# 4. Логи (первый старт качает Qwen2.5-VL-7B ~16 ГБ — это разово)
docker compose logs -f webapp
```

Открыть **`http://<host>:7860`**, загрузить видео, скачать CSV.

Остановить: `docker compose down` (кэш модели сохраняется в volume `hf-cache`).

> Проверка GPU-доступа из контейнера:
> `docker compose exec webapp python -c "import torch;print(torch.cuda.is_available())"`
> → должно вывести `True`.

## Запуск без Docker

```bash
cd mlcore && pip install -r requirements.txt && cd ..
pip install -r webapp/requirements-webapp.txt

export LENTA_YOLO_WEIGHTS=mlcore/runs/yolo/v3/weights/best.pt
export LENTA_LORA_ADAPTER=mlcore/runs/lora/v3/final
python webapp/app.py            # → http://localhost:7860
```

(Без указания env берутся пути по умолчанию внутри `mlcore/runs/...`.)

## Использование интерфейса

1. **Загрузите видео** (`.mp4/.mov/.avi/.mkv`) со стеллажа.
2. Выберите режим:
   - **Полный** — детекция + VLM + штрихкод (полный CSV);
   - **Быстрый (smoke)** — только детекция (быстрая демонстрация / проверка I/O).
3. Нажмите **«Распознать»**. В процессе виден прогресс, время и хвост лога.
4. По завершении: скачиваемый **CSV**, **превью таблицы** (до 200 строк),
   **сводка** (число ценников/треков, статистика штрихкодов, время) и
   **полный лог пайплайна**.

Прогоны сериализуются (`concurrency_limit=1`) — GPU обслуживает одно видео
за раз; остальные ждут в очереди.

## Формат выходного CSV

29 колонок строго в порядке эталона организаторов (см.
`mlcore/src/inference/csv_writer.py::COLUMNS`):
`filename, product_name, price_default, price_card, price_discount, barcode,
discount_amount, id_sku, print_datetime, code, additional_info, color,
special_symbols, frame_timestamp, x_min, y_min, x_max, y_max,
qr_code_barcode, price1_qr…price4_qr, wholesale_level_1/2_count/price,
action_price_qr, action_code_qr`.

Правила заполнения (по ТЗ организаторов): поле физически отсутствует на
ценнике → `нет`; должно быть, но не распозналось → пустая строка.
`filename` формируется как `<стем>/<имя_файла>` (формат эталона).

## Ограничения и масштабирование

**Ограничения подхода:**
- Qwen2.5-VL-7B (bf16) требует ~20 ГБ VRAM — нужен GPU-сервер; на бесплатном
  CPU-хосте полный режим не потянет (для демо есть smoke-режим).
- Пайплайн загружает VLM на каждый прогон видео (через `pipeline.run()`) —
  это десятки секунд оверхеда на загрузку модели для коротких видео.
- Сильное перекрытие/блики/съёмка сквозь стекло снижают читаемость —
  по ТЗ нечитаемое поле остаётся пустым (не штрафуется как ошибка).
- Один процесс обслуживает один прогон за раз (GPU-сериализация).

**Идеи по масштабированию:**
- **Persistent-модель**: держать VLM/YOLO загруженными в памяти между
  запросами (вынести инициализацию `VLMReader` из `run()` в синглтон) —
  убирает оверхед загрузки, ×N throughput на коротких видео.
- **Очередь задач** (Celery/RQ + Redis) и пул GPU-воркеров вместо
  in-process потока — горизонтальное масштабирование на несколько GPU.
- **rknn (int8)** / квантизация VLM — запуск на edge-устройстве робота
  (приветствуется организаторами), снятие зависимости от тяжёлого GPU.
- Стриминговая обработка кадров с робота (а не пост-фактум по файлу) для
  near-real-time контроля полки.
- Шардирование одного длинного видео по сегментам между воркерами с
  последующим merge треков по границам.
